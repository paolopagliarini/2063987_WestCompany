Mission briefing: “Welcome to Mars. Please Don’t Die.”

We are in 2036. After a 12-hour shift of questionable architectural decisions at SpaceY, you are
“promoted” to Mars Operations by being accidentally shipped to Mars while sleeping at your desk.
You wake up in a fragile habitat whose automation stack is partially destroyed. Devices speak
incompatible dialects. Some of them stream telemetry. Others respond only to polling. Actuators
are still reachable — if you can invoke them correctly.
Your mission: rebuild a distributed automation platform capable of ingesting heterogeneous
sensor data, normalizing it into a unified internal representation, evaluating simple automation
rules, and providing a real-time dashboard for habitat monitoring.
Failure means thermodynamic consequences.

3.1

How to run the simulator

To start the simulator, first load the image from the provided tar archive and then run the container
by publishing port 8080:
docker load -i mars-iot-simulator-oci.tar
docker run --rm -p 8080:8080 mars-iot-simulator:multiarch_v1

All endpoints are served from the base URL http://localhost:8080. The OpenAPI documentation is available at http://localhost:8080/docs and http://localhost:8080/openapi.json.
You can also verify the simulator and list available devices using:
curl -s http://localhost:8080/health
curl -s http://localhost:8080/api/discovery | jq

3.4

Actuator APIs

Actuator state can be inspected via the actuators endpoint:
curl -s http://localhost:8080/api/actuators | jq

To change a state, POST a JSON payload to the actuator resource, for example:
curl -X POST http://localhost:8080/api/actuators/cooling_fan \
-H ’Content-Type: application/json’ \
-d ’{"state":"ON"}’

Available actuators include:
• cooling_fan
• entrance_humidifier
• hall_ventilation
• habitat_heater
Actuators are controlled exclusively via REST.

4

What you must deliver

You must design and implement a distributed automation platform that:
3

1. Collects data from simulated devices (polling and/or stream depending on team size)
2. Normalizes heterogeneous payloads into a standard internal event format. You need to
define and document such a standard, and provide a file named input.md as support for your
solution.
3. Uses an event-driven architecture internally (message broker required)
4. Evaluates simple event-triggered automation rules
5. Maintains and exposes the latest state of each sensor (in-memory caching)
6. Provides a real-time dashboard. You must provide the user stories in input.md.

4.1

Persistence

• Historical sensor data persistence is not required.
• The latest state per sensor may be cached in memory.
• Automation rules must be persisted, so that they survive service restarts.
Rule persistence can be implemented using any kind of relational database (e.g., Postgres), noSQL
database (e.g., MongoDB), or embedded database (e.g., SQLite). Complex schema design is not
required; a simple rule table/schema is sufficient.

4.2

Authentication

Authentication is not required. There are no user accounts. The platform may be treated as
single-tenant. All sensors are globally visible: there is no concept of ownership or association.

5

Functional requirements

5.1

Device ingestion and normalization

The system must:
• Ingest device data
• Convert incoming data into a unified internal event schema

5.2

Automation engine

The automation engine must support simple if–then rules.
IF <sensor_name> <operator> <value> [unit]
THEN set <actuator_name> to ON | OFF
Supported operators: <, <=, =, >, >=.
Example: IF greenhouse_temperature > 28 °C THEN set cooling_fan to ON
4

Rules must:
• Be evaluated dynamically on event arrival
• Be persisted
• Trigger an actuator state update when condition is met

5.3

Frontend dashboard

The frontend must provide a real-time monitoring dashboard. A widget-based layout may be used,
but it is not mandatory: static dashboards are also acceptable.
Optional widget approach
• Widgets can be configurable (select sensor binding).
• Example widget types include live value, line chart (updates while the page is open), actuator
toggle, or custom visualizations by data type.
The frontend must also provide a rule management interface and real-time updates via WebSocket
or SSE.

6

Architectural constraints

Mandatory
• Use at least one message broker, e.g., ActiveMQ, Kafka, etc.
• Use multiple backend services (avoid a single monolithic service)
• Separate ingestion, processing, and presentation
• Persist automation rules
• Provide architecture diagrams
Strongly discouraged: Tight coupling between services
